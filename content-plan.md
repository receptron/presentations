# MulmoClaude プレゼン・映像シリーズ 制作プラン

## 1. 目的とゴール

- 10 月末をマイルストーンに、MulmoClaude の魅力(What)・使い方(How)・ビジョン(Why)を伝える多言語プレゼン・映像シリーズをこのリポジトリで制作・発信する。
- ゴールは「なくてはならないツール」と感じるヘビーユーザーを増やすこと。ビジネス指標としては Daily Active User(DAU)。
- 分担: **What / How = 西井、Why = 中島さん**。
- 特に重要なテーマは「LLM に指示しながら、リッチな機能・ビューを持つコレクションを複数の段階を経て作っていく過程」を描く How コンテンツ。
- スクリーンショット・画面キャプチャを使い、見ただけで魅力・使い方が理解できるようにする。
- 実ユーザーの応用例を紹介する形式も What コンテンツとして扱う。
- 多国語化を強く意識する。画面キャプチャ・録画は 1 言語だけなら英語、必要であれば日英両方を用意する(§4)。
- 機能紹介の X 投稿に 10 秒程度の画面録画を添える軽量アウトプットも並行する(§3)。
- ウェビナー・イベント登壇は、まず制作体制が回り始めてから(§9)。

### このプランの位置づけ

- 走り出す前に、コンテンツ戦略・KPI・制作パイプライン・スケジュールを 1 枚にまとめ、チームで方向性を合意するためのたたき台。
- script の叩き台 5 本(`mulmoclaude/tutorials/`・`mulmoclaude/demos/`)で議論を具体化する。

## 2. 成功指標(KPI)のたたき台

10 月末時点で何をもって「うまくいった」とするかを、事前に数値で合意しておくためのたたき台。
ファネルの 4 層で提案する(数値はすべて仮)。

| 層 | 指標 | たたき台の目標(10 月末) | 計測手段 |
|---|---|---|---|
| 制作 | 公開本数 | 12 本(What 6 / How 6)、各 2 言語以上(ja/en)。Why(中島さん担当分)は本リスト外 | このリポジトリ |
| 制作(軽量) | X 向け 10 秒クリップ提供 | 週 1〜2 本を SS の X 運用に供給 | 提供ログ / X の反応 |
| 到達 | 動画視聴数・インプレッション | 合計 5,000 視聴 | YouTube Analytics / SNS |
| 転換 | 新規セットアップ数、GitHub stars、npm DL | stars +100、npm DL 推移の右肩上がり | GitHub / npm stats |
| 定着 | DAU / WAU | ヘビーユーザー 10 人 | **要相談: 計測手段が現状ない** |

### 論点

- 最重要指標は DAU だが、MulmoClaude に利用状況テレメトリは現状存在しない。
  - opt-in の匿名起動カウント(ping)を入れるか、コミュニティでのヒアリング・自己申告で代替するかを要相談。
  - 動画側でコントロールできるのは「到達」まで。評価は「制作 + 到達」を主、「転換・定着」を参考値とする整理もあり得る。
- アウトプットの量・質が外から見えて計れる状態を作る。週 1 本公開 + 週次の数字報告(曜日固定)がその実装。

### コアセットと見せる順(2026-07-12 追記)

- 12 本は 10 月末までの制作パイプラインの本数であり、視聴者に見せるセットの数ではない。ローンチ(X / YouTube / Product Hunt)は**コアセット 3〜4 本**で行う。

- 制作順: What(A: D1 上書き)を最優先で完成 → How(ほぼ完成)→ Why(中島さん進行中)。
- 残りの D/T は公開後の**週次コンテンツ**として反応を見て着手する。見せる順は A → D3(collections-showcase)が自然: A が「何ができるか」(広く浅く・実例)、D3 が「暮らしの中でどうつながるか」(1 ジャーニーを深く)の続編。

## 4. 制作パイプライン

1 本あたりの流れ(1 サイクル 3〜5 営業日を想定):

1. **script 作成** — 既存 script の params ブロック(テーマ・話者・canvasSize)を流用し、beats を書く。
2. **validate** — `mulmoScriptSchema.safeParse` でレンダリング前に検証(CLAUDE.md 記載の手順、`scripts/validate-all.js`)。
3. **キャプチャ収録** — デモ用のクリーンな workspace(ダミーデータ)を用意し、スクリーンショット・画面録画を撮る。該当 beat のスライドプレースホルダを実キャプチャに差し替える。
4. **レンダリング** — `npm run movie -- <file>`(`-g` で output/ にグルーピング)。`--estimate` でコスト確認。
5. **レビュー** — Slack でレビュー → 修正。
6. **多言語展開** — mulmocast の翻訳パイプライン(`-l <lang>` / `-c <lang>`)で ja/en から展開。MulmoViewer 用の多言語プレゼンも同じ script から出す。
7. **配信** — YouTube 公開・MulmoViewer ホスティング・SNS 告知。

### キャプチャ体制(要準備)

- デモ専用のクリーン workspace を 1 つ用意し、個人情報を含まない**英語のダミーデータ**で再現可能にする。
- **言語方針**: キャプチャ・画面録画は 1 言語だけなら英語で収録する。日本語版が必要なコンテンツは日英 2 テイク収録する。
- mulmocast に「言語で画像を切り替える」機能が入れば、1 script で日英キャプチャを出し分けられる(有本さんに要望中。実装されたら 2 テイク運用から切替運用に移行)。
- 録画: macOS 標準の画面収録で足りるか、まず 1 本作って判断。
- キャプチャは差し替え前提なので、script は先にスライドプレースホルダで全 beats を完成させ、ナレーションと尺を固めてから収録する(録り直しを最小化)。
- 収録のついでに X 向け 10 秒クリップを切り出す(§3 軽量フォーマット)。
